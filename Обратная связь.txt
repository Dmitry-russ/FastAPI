Для начала сделать не корень, а 
/creatures/...   а затем твои url
Потому что:
1. По Url не понятно с какой сущностью мы общаемся
2. если тебе нужно будет дополнить это API то могут возникнуть проблемы

Паттерн "/{someone}/" кажется хорошей идеей, но на самом деле это грозит серьезными проблемами.
Посмотри пример с another_url и another_url_2

В данном случае рекомендую переделать либо на 
/creatures/get/{someone}/ -- но тут тоже могут быть проблемы с url

/creatures/ а класс передавать у же как параметр запроса
как-то так http://127.0.0.1:8000/creatures/?creature_kind=dragon
Мне второй вариант нравится больше

Про параметры хорошая статья https://progressivecoder.com/how-to-perform-fastapi-query-parameter-validation-using-query-function/#4-regex-validation


Т.е. для того чтобы у тебя получился проект в портфолио в нем нужно доделать:
1) Красиво оформленное понятное API
2) Еще раз посмотри ответы, может быть сделать более подробные сообщения
3) По данным можно оставить так, но лучше сделать еще и вариант с настоящей БД.
При этом текущий вариант не убирай, а просто сделай варианта запуска программы (когда работает со структурами, когда с БД)
